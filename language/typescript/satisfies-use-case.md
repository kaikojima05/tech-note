## 全部 satisfies 演算子で良くない？

型アノテーション、型アサーションと satisfies 演算子を比較した際にふと思った。
<br />「これ全部 satisfies 演算子でよくね？」
<br />特に型アサーションは不具合の温床になりやすいので、satisfies 演算子と型アサーションを使用すべき境界線を調べた。

## 原則: satisfies 演算子を使用すれば問題ない

TypeScript の公式ドキュメント（サバイバル TypeScript, TypeScript Deep Dive）では**「極力型アサーションを使用しないこと」**を明記している。
<br />そのため、型アノテーションや型アサーションが必要と感じた場面では、**原則、`satisfies 演算子` を使用して問題ない**。
<br />特に、型アサーションを使用すべきではない理由は、上記にある通り**不具合の温床になりやすい**からであるが、それは下記の理由からである。
<br />
<br />1. 型アサーションはプログラマがコンパイラに型の推論結果を矯正するもの（厳密には推論結果の上書き）である
<br />2. number 型の値を string 型に矯正させる等の**「お互いの型に共通する点が少なすぎる場合」**を除き、型アサーションを使用するとリンターや LSP が微妙な型エラーを検出できなくなり、本来ビルド時点で気づけたような不具合が隠匿されてしまう
<br />3. 多くの場合、コンパイラに型を推論させる方が可読性・保守性に優れる
<br />
<br />ちなみに、関係ないが型アサーションは実行時に値の型変換を行うわけではなく、コンパイラに型を伝えているだけなので、`キャスト` ではなく `アサーション` と呼ばれる。

## とはいえ...

下記のケースでは **`as（型アサーション）`** を使用することが推奨される。
<br />
<br />1. DOM や API 等、外部からデータを取得してきた場合（例: getElementById({id}) 等）
<br />2. `JSON.parse()` が返す any の結果
<br />3. ※かなり局所的ではあるが、ライブラリの型定義が間違っている時など
<br />
<br />ただし**型アサーションは `嘘` をつくことができる機能**なので、まずは satisfies 演算子を使用して解決できないかを試す。
